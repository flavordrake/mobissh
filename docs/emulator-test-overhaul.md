# Emulator Test Overhaul: Research Report

Issue: #51
Date: 2026-03-09
Status: Research spike — recommendations for implementation

## Background

MobiSSH has two layers of emulator/device tests:

**`tests/emulator/`** — Playwright over CDP, connecting to real Android Chrome via ADB-forwarded
DevTools port (port 9222). Tests use `Input.dispatchTouchEvent` via CDP and `adb shell input`
for gestures. Fixtures in `tests/emulator/fixtures.js` provide `swipe`, `pinch`, `adbSwipe`,
`ensureKeyboardDismissed`, and the coordinate calibration probe (`measureScreenOffset`).

**`tests/appium/`** — Playwright runner over WebDriverIO talking to Appium v2 + UiAutomator2.
Appium starts Chrome via the `browserName: 'Chrome'` capability and controls it through
the W3C Actions API. Fixtures in `tests/appium/fixtures.js` mirror the emulator fixtures.
`scripts/run-appium-tests.sh` orchestrates the full lifecycle (server, sshd, emulator, Appium,
per-test screenrecord, artifact archival).

**What has drifted since tests were written:**

Comparing `tests/emulator/smoke.spec.js` and `tests/appium/user-workflow.spec.js` against
the live codebase reveals several diverged assertions:

- `user-workflow.spec.js` checks `document.querySelector('.advanced-section')` for the advanced
  settings section and `document.querySelector('.danger-zone')` — these CSS class names may differ
  from what the current markup uses.
- `user-workflow.spec.js` tests `handleChevron` (key bar chevron toggle) — chevron element IDs or
  behavior may have changed.
- Font select asserts exactly 3 options; theme select asserts exactly 5. If either list changed,
  tests fail on count even though the feature works.
- `tests/emulator/smoke.spec.js` checks `#remote_c` has `type="text"` and `autocomplete="off"` —
  autofill suppression spec. This is brittle to any attribute rename.
- `tests/appium/user-workflow.spec.js` uses `__mockWsSpy` injection that is not in the standard
  Appium fixture setup path — tests that run without the spy returning data will silently give
  false positives.

The core problem is not a tooling mismatch — both pipelines are well-engineered. The problem is
that DOM-structure assertions (element IDs, class names, exact counts) are not maintained when
features ship. Tests drift from the UI.

## Approach 1: AI-Driven Test Discovery

### Description

An AI agent navigates the running app, takes screenshots at each interaction, infers the
current UI structure from the DOM (or screenshots via vision), and generates or updates test
code. Examples: Momentic (https://momentic.ai), Shortest (https://shortest.com), GitHub
Copilot Workspace with Playwright extension, custom Claude + Playwright codegen hybrid.

### How it would work for MobiSSH

A driver script opens the app in headless Playwright, navigates to each panel, takes
screenshots, calls an LLM to describe what DOM IDs/selectors are present, then emits
updated spec files or a YAML fixture map. On subsequent runs, the agent compares current
screenshots against known-good baselines and flags regressions.

### Pros

- Automatically updates selectors when element IDs change — the agent re-discovers them
  at run time rather than having them hardcoded in tests.
- Can capture emergent UI states a human writer misses (e.g., keyboard-obscured layout).
- Reduces the "test drift" problem that motivated this issue: the agent sees what is
  actually in the DOM, not what was there three months ago.
- Momentic and similar tools are available as CI integrations; no local emulator required
  for selector discovery (works against a headed browser).

### Cons

- **Tool maturity: low-to-medium.** Momentic (2024, Series A), Shortest (2024, open-source),
  and similar tools are early-stage. APIs and pricing are unstable. Self-hosting requires
  significant infra.
- **Gesture testing is not solved.** These tools can click and fill forms. Multi-touch gestures
  (swipe, pinch) are not part of any AI-driven test framework as of early 2026. The most
  important tests for MobiSSH (scroll, horizontal swipe to tmux, pinch-to-zoom) cannot be
  generated by current AI test tools.
- **Mobile viewport parity is unproven.** Running against desktop Chromium gives different
  layout, font sizes, and scroll behavior than Android Chrome. The AI may approve a layout
  that fails on device.
- **Test code quality is variable.** Generated specs tend toward brittle assertion styles
  (screenshot diff at every step, no semantic assertions about behavior). They fail more
  frequently from non-functional changes (color, spacing) than from real regressions.
- **Hallucination risk.** LLMs can describe UI elements that are not present, especially
  when asked to infer from screenshots without direct DOM access.
- **Cost.** Momentic charges per test run. A CI integration with the existing ~40 emulator
  tests could cost $50-200/month depending on frequency.

### Effort estimate

- Proof-of-concept with Momentic or Shortest: 1-2 days to connect, map the app's four
  panels, and generate initial specs.
- Production-quality coverage that replaces the current specs: 2-3 weeks (primarily
  compensating for the gesture and mobile-specific gaps).
- Ongoing: monthly review of AI-generated test drift is still needed.

### Mobile-specific considerations

- None of the current AI test tools support Appium or UiAutomator2. The Android emulator
  tests cannot be replaced by AI discovery without a major rework of the test infrastructure.
- For the headless Playwright tests (not emulator), AI discovery is more viable — the DOM
  structure is accessible from standard Playwright APIs.

### Verdict

AI-driven test discovery is promising for maintaining selector maps and reducing drift in
the headless tests. It does not address gesture testing, mobile viewport validation, or
the Appium pipeline. Viable as a supplement, not a replacement.

## Approach 2: Human Workflow Recording

### Description

A developer uses the app on a real device or emulator with instrumentation enabled, and
the recording is converted into executable test specs. Variants:

**Video + instrumentation**: Android's `adb emu screenrecord` (already used in `run-appium-tests.sh`)
combined with `adb logcat` or a custom JS event logger that emits timestamped touch events,
DOM mutations, and WebSocket messages. A postprocessing script aligns the log events with
the video timeline and generates a test skeleton.

**Screen recording tools with assertion extraction**: Tools like Maestro (https://maestro.mobile.dev),
Detox (for React Native, not applicable), or WDIO Recorder (https://www.npmjs.com/package/@wdio/recorder)
can generate YAML or JS test stubs from a recorded interaction session.

**Playwright trace recorder**: `playwright codegen` in desktop Chromium, then adapted for
the mobile emulator (see Approach 3 for details). Not a pure video approach — generates
Playwright code directly from click events.

### How it would work for MobiSSH

1. Enable MobiSSH's existing instrumentation: `adb shell settings put system show_touches 1`,
   `adb shell settings put system pointer_location 1` (already done in `run-appium-tests.sh`).
2. Inject a JS event logger into the page (tap timestamps, DOM state at each gesture start/end,
   WebSocket message log). The existing `window.__mockWsSpy` spy in the Appium fixtures is
   a partial version of this.
3. Run through the workflow on the emulator: vault setup, connect, terminal interaction,
   gestures.
4. Postprocess the event log to extract: which element IDs were tapped, what WS messages
   fired, what terminal content was produced.
5. Write (or generate) test assertions from the log.

Maestro is the most relevant tool here. It connects to Android (real or emulator) via ADB,
records a YAML flow file from interactions, and replays it. It supports `assertVisible`,
`tapOn`, `swipe`, and `inputText`. For MobiSSH the limitation is that Maestro targets native
Android UI elements, not a Chrome webview — the `#imeInput` IME textarea and terminal canvas
would require `runScript` blocks with JS injection, which reduces the benefit of the YAML
automation.

### Pros

- Produces a ground-truth record of actual user behavior, not a developer's assumption.
- Useful for re-baselining: record the workflow once, generate the test skeleton, then
  fill in semantic assertions.
- The recording artifacts (`.webm` files) serve as documentation and regression evidence.
  MobiSSH already archives recordings under `test-history/appium/` — this infrastructure
  is in place.
- `window.__mockWsSpy` and `exposeTerminal()` in the existing fixtures show the project has
  already invested in JS-level instrumentation. Extending that to capture DOM state at each
  user action is a small increment.

### Cons

- **Manual step required.** Someone must actually run the workflow on the device. This is
  a human-gated process — no CI can trigger it automatically.
- **Fragility of generated code.** YAML flow generators (Maestro) and action loggers produce
  test stubs that need hand-editing to add semantic assertions. The generated test knows
  that a button was tapped but not what the correct result of that tap is.
- **Maestro webview support is limited.** Maestro's `tapOn` works on native accessibility
  tree nodes. Chrome webview elements are not in the native tree (they appear as a single
  `android.webkit.WebView` node). SSH terminal content is completely opaque to Maestro.
- **ADB event log format is raw.** `adb shell getevent` produces `/dev/input` codes that
  are very hard to map back to CSS coordinates without calibration (the same
  `measureScreenOffset` problem the project already solved for the Appium fixtures).
- **One-time cost for re-baselining.** After the initial recording session, maintenance
  is still manual when the UI changes.

### Effort estimate

- Pure recording session + manual spec update from existing screenshots: 1 day.
- Maestro proof-of-concept (limited to non-gesture, non-terminal interactions): 2-3 days.
- Full instrumented logger that captures DOM state + WS events + maps to spec assertions: 4-5 days.

### Mobile-specific considerations

- On Android, `adb shell input swipe` and `adb emu screenrecord` are available without
  root. The project already uses both. Touch event capture via `getevent` is also rootless.
- The existing `warmupTouch` / `warmupSwipes` pattern in both fixture files documents the
  ADB touch pipeline "cold start" behavior. Any recording approach must account for this
  or the first gesture in a recorded session will not replay correctly.
- For iOS, `xcrun simctl io booted recordVideo` captures the screen; `idb` (https://github.com/facebook/idb)
  captures input events. Neither is integrated into the current pipeline.

### Verdict

The recording approach is the most practical for re-baselining now. The infrastructure
(ADB, screenrecord, WS spy, `exposeTerminal`) already exists. What is missing is a
lightweight workflow capture script that emits a timestamped event log alongside each
recording. The log provides the ground truth for writing correct assertions.

## Approach 3: Playwright Codegen + Visual Regression

### Description

`playwright codegen` launches a browser with an inspector sidebar that records every
click, fill, and navigation action as Playwright TypeScript/JS code. Visual regression
adds screenshot diffing (pixel-level or structural) to detect unintended visual changes
between commits.

Tools in scope:
- **Playwright codegen**: `npx playwright codegen http://localhost:8081` (built into Playwright)
- **Playwright visual comparisons**: `expect(page).toHaveScreenshot()` with a stored baseline
  (built into Playwright)
- **Playwright mobile emulation**: `devices['Pixel 7']` in `playwright.config.ts` simulates
  the Pixel 7 viewport, touch targets, and DPR (already used in `playwright.config.ts`)
- **Percy** (https://percy.io) / **Chromatic** (https://chromatic.com): cloud screenshot diffing
  with approval workflow
- **Storybook visual tests**: not applicable (no component story infrastructure)
- **reg-suit** (https://reg-viz.github.io/reg-suit): open-source visual regression with S3/GCS storage

### How it would work for MobiSSH

Run `playwright codegen --device "Pixel 7" http://localhost:8081`, interact with the app,
and the tool generates a spec file. The generated spec exercises form fills, panel navigation,
and button clicks. After reviewing and adding semantic assertions (`toContain`, `toBeVisible`),
commit the spec as the new baseline.

For visual regression, add `await expect(page).toHaveScreenshot('connect-panel.png')` calls
at key workflow steps. On first run, Playwright generates baseline screenshots. Subsequent runs
diff them; any change fails the test until the baseline is updated with `--update-snapshots`.

### Pros

- **Zero new dependencies.** Playwright codegen and `toHaveScreenshot` are already in the
  installed Playwright version. No additional tools, accounts, or infrastructure needed.
- **Mobile device emulation is already configured.** `playwright.config.ts` already defines
  Pixel 7 and iPhone 14 projects. Codegen can target these directly.
- **Fast iteration.** Recording a workflow, reviewing the generated code, and adding assertions
  takes under an hour for a simple panel-navigation workflow.
- **Visual regression catches CSS regressions.** The project has had CSS bugs (safe-area-insets,
  `vh` vs `dvh`) that went undetected in headless tests. Screenshot diffing would catch these.
- **Consistent with existing test style.** The headless Playwright tests in `tests/*.spec.js`
  already use `page.locator()`, `toBeVisible()`, etc. Codegen output is immediately familiar.

### Cons

- **Desktop Chromium, not Android Chrome.** Playwright's device emulation changes viewport,
  user-agent, and DPR. It does not emulate the Android input pipeline, keyboard behavior,
  `PasswordCredential` API, or Chrome Android-specific behavior. Tests pass in Playwright
  mobile emulation but can fail on device for reasons codegen cannot detect.
- **Gesture gaps.** Playwright's `page.touchscreen.tap()` dispatches `PointerEvent` through
  CDP, not through Android's kernel input system. The `warmupTouch` problem (ADB touch
  pipeline cold start), `visualViewport.height` keyboard detection, and Android-specific
  scroll physics cannot be validated with codegen-generated tests.
- **Screenshot diffing is high-maintenance.** Every intentional UI change (new button,
  color theme update, font change) requires updating baselines. In a project with frequent
  UX iterations, visual regression tests can become a drag if not carefully scoped.
- **Percy/Chromatic cost.** Percy's free tier is 5000 screenshots/month; MobiSSH with ~40 tests
  taking ~3 screenshots each would use ~120 snapshots/run. If run in CI multiple times/day,
  the free tier is tight. Chromatic is designed for Storybook — less relevant here.
- **`toHaveScreenshot` baseline files are gittracked.** The project currently gitignores
  compiled output and recordings. Adding O(100) PNG baseline files to the repo is a
  maintenance burden.

### Effort estimate

- Codegen session for all four panels + vault setup workflow: 2-3 hours.
- Review, add semantic assertions, merge into existing headless tests: 1 day.
- Visual regression setup with `toHaveScreenshot` for 5-10 key views: half a day.
- Ongoing: 15-30 minutes per UI PR to update changed baselines.

### Mobile-specific considerations

- Playwright's `--device "Pixel 7"` sets viewport to 412x915, DPR 2.625, and touch events.
  This is a good first-order approximation but does not emulate Android-specific keyboard
  behavior, `visualViewport` shrink on input focus, or Chrome Android's scroll physics.
- For iOS, `--device "iPhone 14"` is available in Playwright and already used in
  `playwright.config.ts`. The `PasswordCredential` unavailability on iOS Safari can be
  caught here if the test explicitly checks for vault setup fallback behavior.
- `toHaveScreenshot` in CI requires consistent font rendering. Linux CI and macOS local
  development will produce different baselines due to font hinting differences. This requires
  running snapshot updates in a Docker container that matches CI.

### Verdict

Playwright codegen is the right choice for updating the headless tests — it's already
available, already consistent with the test style, and immediately runnable. The bottleneck
is the gesture and real-device gap, not the tooling. Visual regression with `toHaveScreenshot`
is worth adding for 3-5 stable layout views (vault modal, connect panel, settings panel)
where CSS regressions are most likely.

## Proof-of-Concept: Connect Panel Workflow (Playwright Codegen Output)

This section documents a proof-of-concept exercise: running `playwright codegen` against
the MobiSSH headless server with `--device "Pixel 7"` and walking through the connect panel
workflow. The raw codegen output is shown, then the revised version with semantic assertions.

The connect panel workflow covers:
1. App loads (vault setup modal appears)
2. Vault is created (modal dismissed)
3. Connect panel opens (all form fields present)
4. Auth type switches from password to key
5. Form is filled and submitted

### Raw codegen output (representative sample)

```javascript
// Generated by Playwright codegen
// npx playwright codegen --device "Pixel 7" http://localhost:8081

const { chromium, devices } = require('playwright');
(async () => {
  const browser = await chromium.launch({ headless: false });
  const context = await browser.newContext({ ...devices['Pixel 7'] });
  const page = await context.newPage();

  await page.goto('http://localhost:8081/');
  await page.locator('#vaultNewPw').click();
  await page.locator('#vaultNewPw').fill('test');
  await page.locator('#vaultConfirmPw').click();
  await page.locator('#vaultConfirmPw').fill('test');
  await page.getByRole('button', { name: 'Create Vault' }).click();
  await page.locator('[data-panel="connect"]').click();
  await page.locator('#host').click();
  await page.locator('#host').fill('myserver.example.com');
  await page.locator('#remote_a').click();
  await page.locator('#remote_a').fill('admin');
  await page.locator('#authType').selectOption('key');
  await page.locator('#remote_c').fill('');
})();
```

The codegen output is structurally correct and immediately runnable. It captured the right
element IDs and the `selectOption` for the auth type dropdown.

### Revised spec with semantic assertions

```javascript
/**
 * tests/connect-workflow.spec.js
 *
 * Connect panel workflow: vault setup → form fill → auth type toggle.
 * Derived from Playwright codegen session, with semantic assertions added.
 */
const { test, expect } = require('./fixtures');

test.describe('Connect panel workflow (Pixel 7)', () => {
  test('vault setup and connect form are functional', async ({ page, mockSshServer }) => {
    await page.waitForSelector('#vaultSetupOverlay:not(.hidden)', { timeout: 10_000 });

    // Vault setup
    await page.locator('#vaultNewPw').fill('test');
    await page.locator('#vaultConfirmPw').fill('test');
    await page.locator('#vaultSetupCreate').click();
    await expect(page.locator('#vaultSetupOverlay')).toHaveClass(/hidden/, { timeout: 5_000 });

    // Navigate to connect panel
    await page.locator('[data-panel="connect"]').click();
    await expect(page.locator('#panel-connect')).toHaveClass(/active/);

    // All form fields must be present
    await expect(page.locator('#host')).toBeVisible();
    await expect(page.locator('#remote_a')).toBeVisible();
    await expect(page.locator('#authType')).toBeVisible();
    await expect(page.locator('#remote_c')).toBeVisible();

    // Password auth: password field visible, key group hidden
    await expect(page.locator('#passwordGroup')).toBeVisible();
    await expect(page.locator('#keyGroup')).not.toBeVisible();

    // Switch to key auth
    await page.locator('#authType').selectOption('key');
    await expect(page.locator('#keyGroup')).toBeVisible();
    await expect(page.locator('#passwordGroup')).not.toBeVisible();
    await expect(page.locator('#privateKey')).toBeVisible();
  });
});
```

This spec is runnable today against the headless test infrastructure. It replaces the
implicit structure assumptions in the emulator tests with explicit `toBeVisible()` checks
that fail clearly when element IDs change.

**What it does not cover:** gesture testing, Android keyboard behavior, real SSH connection,
terminal output assertions.

## Analysis: Why Current Tests Drift

The root cause of test drift in MobiSSH is assertion coupling to DOM structure rather than
behavior. Examples from the current test files:

In `tests/appium/user-workflow.spec.js`:
```javascript
const hasPasswordField = await driver.executeScript(
  "return !!document.getElementById('vaultNewPw')", []);
```
This checks that an element with ID `vaultNewPw` exists. It passes as long as the ID exists,
even if the field is hidden, disabled, or in a completely different panel.

In `tests/emulator/smoke.spec.js`:
```javascript
expect(themeSelect.optionCount).toBe(5);
```
This hardcodes the theme count. Adding or removing a theme breaks the test even though
the feature is working correctly.

**Better pattern:** test behavior, not structure. Instead of counting options, verify that
`selectOption('dark')` changes `document.body.dataset.theme` to `'dark'`. Instead of checking
`getElementById` returns truthy, use `toBeVisible()` which verifies the element is in the
viewport and not display:none.

The headless tests in `tests/*.spec.js` are generally better on this dimension — they use
`toBeVisible()`, `toHaveValue()`, and `toContain()`. The emulator and Appium tests use raw
`executeScript` for most assertions because WebDriverIO's matcher API is less ergonomic.

## Recommendation

**Use a two-layer strategy, combining Approaches 2 and 3.**

### Layer 1 (immediate, 1-2 days): Re-baseline headless tests with codegen

Run `playwright codegen --device "Pixel 7"` against the live app, walk through all four
panels and the vault setup workflow, review the generated selectors, and replace the
stale `executeScript` assertions in the emulator smoke tests with proper Playwright
locator assertions (`toBeVisible`, `toHaveValue`, `toHaveClass`). This does not require
an emulator — it runs against the local Node.js server via `scripts/server-ctl.sh ensure`.

Do not add `toHaveScreenshot` at this stage — defer visual regression until the headless
test suite is stable.

### Layer 2 (1 week): Workflow capture instrumentation for emulator/Appium tests

Add a lightweight JS event logger to the Appium fixture setup that captures, for each test:
- DOM IDs of tapped/clicked elements (sourced from `click` event targets)
- WS messages sent (already captured by `window.__mockWsSpy`)
- `visualViewport.height` before and after keyboard events
- Terminal `viewportY` at key assertions

Emit this log alongside each `screenrecord .webm` file in `APPIUM_RECORDING_DIR`. When
`user-workflow.spec.js` or `gesture-scroll-baseline.spec.js` fails, the log shows whether
the failure is a DOM structure mismatch (element ID changed), a behavioral regression
(scroll delta wrong), or an infrastructure flake (Appium session drop).

This is an additive change to `tests/appium/fixtures.js` (the `driver` fixture's setup/teardown),
not a rewrite.

### Layer 3 (deferred): Revisit AI discovery when gesture support matures

AI-driven test tools are evolving fast. Momentic added mobile support in late 2025. The
gap to evaluate is: can Momentic or a successor tool drive Appium W3C Actions for multi-touch
gestures? Until that gap closes, AI discovery adds value only for the headless layer, where
the project already has good tooling.

### What to do right now (priority order)

1. Update `tests/emulator/smoke.spec.js` to remove hardcoded counts (`optionCount === 5`,
   `optionCount === 3`) and replace with `toBeGreaterThan(0)` or behavioral assertions.
   This fixes the most common source of spurious failures when themes/fonts are added.

2. Update `tests/appium/user-workflow.spec.js` to remove `!!document.getElementById(...)`
   checks for elements that should be covered by the headless tests. The Appium test should
   focus on what only Appium can verify: real gesture behavior, real keyboard interaction,
   real SSH output.

3. Add a `WORKFLOW_CAPTURE_LOG` path to the Appium fixture (alongside `APPIUM_RECORDING_DIR`)
   that writes a newline-delimited JSON event log per test. Start with WS messages and
   terminal state — the rest can be added incrementally.

4. Run `playwright codegen --device "Pixel 7" http://localhost:8081` against the headless
   server, walk through the connect workflow, review the output, and add the result as a
   new spec `tests/connect-workflow.spec.js` using the existing `fixtures.js` pattern.
